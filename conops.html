2. Концепция функционирования

Участники (заинтересованные стороны и роли)

  1. Инициатор — сотрудник, который создаёт документ (заявку, договор, служебную записку) и запускает его по маршруту согласования.
  2. Согласующий — руководитель или ответственный сотрудник, который должен утвердить или отклонить документ.
  3. Юрист — специалист юридического отдела, проверяющий документы на соответствие законодательству и внутренним нормативам.
  4. Бухгалтер — сотрудник финансового отдела, работающий с заявками на оплату и финансовыми условиями договоров.
  5. Администратор — специалист по информационным технологиям или кадрам, настраивающий маршруты, права доступа, справочники и шаблоны.
  6. Контрагент — внешняя организация (сторона по договору), взаимодействует через оператора электронного документооборота.

Системные элементы (типовая конфигурация)

  1. Маршрутизатор — ядро системы, управляющее движением задач по ролям и контролем сроков согласования. Определяет последовательность шагов маршрута на основе типа документа и шаблона, настроенного Администратором.

  2. Репозиторий документов — хранилище файлов с поддержкой версионности. Хранит историю изменений, контрольные суммы каждой версии (алгоритм хеширования семейства «Безопасный Алгоритм Хеширования» с длиной 256 бит) и связь версий с карточкой документа.

  3. Модуль электронной подписи — интеграция с криптопровайдером (удостоверяющим центром) для наложения электронных подписей. Поддерживает два вида подписи:

    • Квалифицированная Электронная Подпись — юридически значимая подпись, выдаваемая аккредитованным удостоверяющим центром;
    • Усиленная Неквалифицированная Электронная Подпись — подпись, подтверждаемая кодом в коротком текстовом сообщении, используемая для внутренних согласований и мобильного доступа.

    Перед подписанием проверяет статус сертификата через протокол онлайн-проверки статуса сертификата. При недоступности криптопровайдера переводит документ в режим ожидания.

  4. Журнал аудита — неизменяемая лента событий для юридической значимости. Каждая запись содержит: идентификатор события, идентификатор пользователя, временную метку, сетевой адрес устройства, тип действия, контрольную сумму файла (если применимо). Записи связаны криптографической цепочкой — каждая запись содержит контрольную сумму предыдущей, что исключает незаметную модификацию или удаление.

  5. Сервис уведомлений — отправка уведомлений по электронной почте, push-уведомлений и коротких текстовых сообщений через внешний шлюз. При недоступности основного канала переключается на альтернативный.

  6. Модуль единого входа — подсистема аутентификации. Интегрируется с корпоративной службой каталогов: при входе пользователя запрашивает у службы каталогов проверку учётных данных и получает в ответ токен, содержащий идентификатор пользователя, список ролей, список групп и время истечения. Все остальные модули платформы доверяют этому токену и не запрашивают пароль повторно. Если корпоративная служба каталогов недоступна, Администратор может переключить модуль в режим ручной аутентификации с фиксацией в журнале аудита.

  7. Справочник организационной структуры — кэшированная база сотрудников, ролей и иерархии. Периодически синхронизируется с системой управления персоналом (по умолчанию 1 раз в сутки). Для определения ролей и замещений Маршрутизатор обращается именно к этому справочнику, а не напрямую к внешним системам. При недоступности системы управления персоналом используется последняя кэшированная версия, задачи назначаются по ролям, а не по конкретным именам сотрудников.

  8. Модуль электронного документооборота — обмен юридически значимыми документами с контрагентами через внешнего оператора электронного документооборота. Обеспечивает отправку подписанных документов, приём встречной Квалифицированной Электронной Подписи контрагента, обработку протоколов разногласий. При недоступности оператора документ переводится в статус ожидания с автоматической отправкой при восстановлении.

  9. Модуль архивации — передача завершённых дел в долговременный архив с сохранением всех версий файлов, подписей, метаданных карточки и ленты аудита. Обеспечивает выполнение требований Федерального закона № 125 «Об архивном деле в Российской Федерации» (хранение 5 и более лет). Передача завершается получением квитанции от архива с идентификатором дела и контрольной суммой.

Сценарий 1. Запуск договора на согласование (Инициатор → Юрист → Согласующий → Контрагент)

Цель

  1. Создать договор с контрагентом, провести юридическую экспертизу и бизнес-утверждение с контролем версий, затем направить контрагенту на подпись через оператора электронного документооборота.

Условия начала

  1. Инициатор авторизован в платформе через Модуль единого входа — получен токен с ролями.
  2. Справочник организационной структуры актуален — известны роли Юриста и Согласующего.
  3. Реквизиты контрагента проверены через систему управления ресурсами предприятия — статус контрагента активен.
  4. Шаблон договора доступен для загрузки.

Основной поток

  1. Инициатор создаёт карточку договора, заполняет реквизиты (контрагент, сумма, предмет, статья расходов) и загружает файл — Версия 1.
  2. Маршрутизатор обращается к Справочнику организационной структуры, определяет маршрут: первый шаг — Юрист.
  3. Юрист получает уведомление через Сервис уведомлений (электронная почта и/или push-уведомление с указанием идентификатора документа, типа, имени инициатора и крайнего срока), открывает задачу, вносит правки, загружает Версию 1.1 с пометкой «Правки юриста».
  4. Репозиторий документов сохраняет обе версии с контрольными суммами, связывая их с одной карточкой.
  5. Юрист нажимает «Согласовано» — система фиксирует визу в журнале аудита (идентификатор пользователя, временная метка, решение) и передаёт задачу Согласующему.
  6. Согласующий просматривает список версий с атрибутами (номер версии, автор, дата, комментарий, контрольная сумма), утверждает файл и накладывает Квалифицированную Электронную Подпись через Модуль электронной подписи.
  7. Модуль электронной подписи вычисляет контрольную сумму файла, отправляет её в криптопровайдер, получает подписанную контрольную сумму и штамп времени, сохраняет подпись в карточке документа. Журнал аудита фиксирует событие: идентификатор подписанта, контрольную сумму, серийный номер сертификата, временную метку.
  8. Модуль электронного документооборота направляет подписанный документ Контрагенту через внешнего оператора и фиксирует в журнале аудита факт отправки и квитанцию оператора.
  9. Контрагент подписывает встречной Квалифицированной Электронной Подписью. Оператор электронного документооборота возвращает квитанцию и подписанный экземпляр. Система проверяет подпись контрагента через протокол онлайн-проверки статуса сертификата, сохраняет экземпляр, уведомляет Инициатора.
  10. Инициатор получает уведомление: «Договор подписан обеими сторонами».
  11. Модуль архивации передаёт завершённое дело в долговременный архив: файлы всех версий, подписи, метаданные карточки, лента аудита. Статус документа меняется на «Архивирован».

Результат

  1. Договор подписан Квалифицированной Электронной Подписью обеими сторонами, все версии сохранены с контрольными суммами, аудиторский след с криптографической цепочкой сформирован, документ передан в архив с квитанцией подтверждения.

Исключения (нештатные ситуации)

  1. Юрист отклонил договор 1.1. Юрист вводит текстовый комментарий с причиной отказа (например, «Неверная форма собственности контрагента»). 1.2. Система фиксирует отказ в журнале аудита, переводит документ в статус «Доработка» и возвращает задачу Инициатору.

  2. Конфликт версий (параллельное редактирование) 2.1. Два участника загрузили свои версии параллельно (Версия 1.1 и Версия 1.2). 2.2. Репозиторий документов фиксирует конфликт. 2.3. Тот, кто нажимает «Согласовать» или загружает версию позже, видит предупреждение о конфликте версий. Операция блокируется до разрешения конфликта: участник должен скачать актуальную версию, объединить изменения и загрузить новую версию.

  3. Сертификат подписанта недействителен 3.1. Модуль электронной подписи проверяет сертификат через протокол онлайн-проверки статуса сертификата. Если сертификат отозван или истёк — кнопка «Подписать» блокируется, предлагается перевыпуск сертификата или делегирование задачи заместителю.

  4. Контрагент отклонил документ 4.1. Оператор электронного документооборота передаёт протокол разногласий. Система возвращает задачу Инициатору/Юристу с комментариями контрагента.

  5. Реквизиты контрагента не найдены в системе управления ресурсами предприятия 5.1. Система блокирует отправку заявки и уведомляет Инициатора с требованием завести контрагента в системе управления ресурсами или клиентской базе.

Режимы деградации

  1. Криптопровайдер недоступен (Модуль электронной подписи не может выполнить подписание) 1.1. Документ переходит в статус «Ожидание подписи (оффлайн)». 1.2. Согласующий может проставить внутреннюю визу без юридической силы. 1.3. Документ ставится в очередь на подписание при восстановлении. 1.4. Переход в резервный режим происходит автоматически.

  2. Корпоративная служба каталогов недоступна — затрагивает Модуль единого входа и Справочник организационной структуры 2.1. Модуль единого входа: уже авторизованные пользователи продолжают работать по действующему токену. Новые входы невозможны, пока Администратор не включит ручную аутентификацию с фиксацией в журнале аудита записи «Резервная аутентификация». 2.2. Справочник организационной структуры: Маршрутизатор использует последнюю кэшированную версию. Задачи назначаются по ролям, а не по конкретным именам сотрудников. 2.3. При восстановлении — автоматическая синхронизация.

  3. Оператор электронного документооборота недоступен 3.1. Документ переводится в статус «Ожидание отправки». 3.2. Автоматическая отправка при восстановлении связи. 3.3. Факт перехода в режим ожидания фиксируется в журнале аудита.

Сценарий 2. Срочное согласование служебной записки (мобильный доступ)

Цель

  1. Согласовать срочную служебную записку (например, на выдачу материальных ценностей) в нерабочее время через мобильное устройство.

Условия начала

  1. Сотрудники вне офиса, используется мобильное приложение или адаптивная веб-версия.
  2. Инициатор авторизован через Модуль единого входа — токен действующий.

Основной поток

  1. Инициатор создаёт служебную записку с мобильного устройства, прикрепляет скан-копии (фото с телефона).
  2. Маршрутизатор через Справочник организационной структуры определяет маршрут: Согласующий (руководитель), Бухгалтер — только если сумма выше лимита.
  3. Согласующий получает push-уведомление через Сервис уведомлений, открывает документ в формате переносимого документа на мобильном устройстве.
  4. Согласующий нажимает «Согласовать» и подписывает Усиленной Неквалифицированной Электронной Подписью через код подтверждения в коротком текстовом сообщении.
  5. Сумма ниже порога бухгалтерии — маршрут завершается.
  6. Инициатор и Бухгалтер (для сведения) получают уведомление.

Результат

  1. Служебная записка утверждена руководителем через мобильный интерфейс, виза зафиксирована в журнале аудита с указанием способа подписания.

Исключения

  1. Отсутствие сетевого соединения / обрыв связи 1.1. Мобильное приложение работает в оффлайн-режиме: сохраняет решение пользователя локально. 1.2. При восстановлении связи — синхронизация с сервером с пометкой в журнале аудита «Подписано в оффлайн-режиме» и фактическим временем решения.

  2. Документ требует Квалифицированной Электронной Подписи, но на мобильном устройстве сертификат отсутствует 2.1. Модуль электронной подписи не находит сертификат. Система переводит задачу в статус «Ожидает подпись с персонального компьютера» с указанием срока (по умолчанию 24 часа) и отправляет напоминание.

Режимы деградации

  1. Сервис уведомлений / шлюз недоступен 1.1. Push-уведомление не доставлено. Платформа повторяет отправку через альтернативный канал (короткое текстовое сообщение → электронная почта).

  2. Модуль единого входа недоступен 2.1. Мобильное приложение использует кэшированный токен, если сессия активна. При истечении токена — вход невозможен до восстановления.

Сценарий 3. Заявка на оплату (Бухгалтерия и контроль лимитов)

Цель

  1. Провести заявку на оплату счёта через бухгалтерию с автоматической проверкой бюджетных лимитов и корректности реквизитов.

Условия начала

  1. Заявка создана на основании утверждённого договора.
  2. В системе настроены бюджетные лимиты по статьям расходов (интеграция с системой управления ресурсами предприятия).

Основной поток

  1. Инициатор заполняет заявку: ссылка на договор, сумма, статья расходов, скан счёта.
  2. Маршрутизатор автоматически запрашивает у системы управления ресурсами предприятия: остаток бюджета по статье, лимит договора, реквизиты и статус контрагента.
  3. Если остаток больше или равен сумме заявки и контрагент активен — заявка направляется Бухгалтеру.
  4. Бухгалтер проверяет реквизиты (идентификационный номер налогоплательщика, код причины постановки на учёт, расчётный счёт, назначение платежа).
  5. Бухгалтер нажимает «Исполнить» или «Передать в платёжную систему».
  6. Система фиксирует оплату и списывает сумму из лимита в системе управления ресурсами предприятия.

Результат

  1. Заявка обработана, лимиты скорректированы, действия зафиксированы в журнале аудита.

Исключения

  1. Превышение бюджетного лимита 1.1. Система блокирует отправку заявки Бухгалтеру. 1.2. Маршрутизатор через Справочник организационной структуры находит вышестоящего руководителя или финансового директора и добавляет его в цепочку согласования (эскалация). 1.3. Заявка переходит в статус «Ожидание бюджета».

  2. Реквизиты контрагента не найдены 2.1. Система блокирует отправку заявки и уведомляет Инициатора с требованием завести контрагента в системе управления ресурсами или клиентской базе.

Режимы деградации

  1. Система управления ресурсами предприятия недоступна 1.1. Модуль проверки лимитов переходит в ручной режим. 1.2. Бухгалтер получает задачу с пометкой «Проверить лимиты вручную». 1.3. Бухгалтер активирует элемент подтверждения «Лимит подтверждён вручную» (берёт ответственность на себя) и проводит заявку. 1.4. В журнале аудита фиксируется: идентификатор бухгалтера, временная метка, пометка «Ручной ввод, система управления ресурсами недоступна». 1.5. Переход в резервный режим происходит автоматически.

Сценарий 4. Коллегиальное согласование (параллельный маршрут и голосование)

Цель

  1. Согласовать стратегический документ (Политику, Положение), требующий одобрения нескольких руководителей одновременно.

Условия начала

  1. Создан документ, требующий визирования: Юрист, директор по персоналу, технический директор, финансовый директор.
  2. Маршрут параллельного согласования настроен Администратором.

Основной поток

  1. Инициатор запускает задачу на параллельное согласование.
  2. Маршрутизатор через Справочник организационной структуры находит всех Согласующих и создаёт им задачи одновременно. Каждому отправляется уведомление через Сервис уведомлений.
  3. Каждый Согласующий видит документ и голосует: «За» / «Против» / «Воздержался» / «За, с замечаниями».
  4. Маршрутизатор агрегирует голоса по настроенному правилу (единогласно / большинство / кворум).
  5. Допустим, Юрист — «Против». Результат: «Не согласовано. Юрист против».
  6. Инициатор получает уведомление с протоколом: решение каждого участника, комментарии, итог.

Результат

  1. Документ отклонён с фиксацией позиции каждого участника в журнале аудита, включая комментарии и временные метки голосования.

Исключения

  1. Тайм-аут ожидания голосов (контроль сроков) 1.1. Сервис уведомлений отправляет напоминания при достижении настраиваемого процента от установленного срока. 1.2. Если к крайнему сроку проголосовали не все участники, система применяет настроенную политику: «Молчание — знак согласия» (голос не подавших считается «За») или «Решение по кворуму» (учитываются только поданные голоса). Результат фиксируется в журнале аудита с пометкой «Применена политика тайм-аута».

  2. Голосование «За, с замечаниями» 2.1. Комментарий прикрепляется к документу в Репозитории документов. 2.2. Если кто-то голосует «Против» с блокирующими правками — задача возвращается Инициатору, остальные голоса помечаются как неактуальные.

  3. Согласующий недоступен, требуется делегирование 3.1. Согласующий может делегировать задачу заместителю. Система проверяет замещение через Справочник организационной структуры, переназначает задачу и фиксирует делегирование в журнале аудита.

Режимы деградации

  1. Справочник организационной структуры устарел (система управления персоналом не отвечает, не удаётся найти заместителей) 1.1. Используется последняя кэшированная версия, задачи назначаются по ролям, а не по конкретным именам сотрудников.

  2. Сервис уведомлений недоступен 2.1. Напоминания по срокам не отправляются. Задачи остаются доступными при входе в платформу. При восстановлении — массовая рассылка накопившихся уведомлений.

Сценарий 5. Аудит и разбирательство (юридическая значимость)

Цель

  1. Восстановить полную картину событий по договору в случае судебного спора или проверки контролирующими органами (Федеральная налоговая служба, внешний аудит).

Условия начала

  1. Юрист или Администратор имеет право на просмотр журнала аудита (права настроены Администратором через управление ролевым доступом).

Основной поток

  1. Юрист открывает карточку спорного договора. Если документ в архиве — Модуль архивации запрашивает его из долговременного хранилища по идентификатору карточки и получает все файлы, подписи и метаданные.
  2. Юрист переходит на вкладку «Аудит». Система отображает неизменяемую ленту из журнала аудита, где каждая запись содержит: идентификатор события, идентификатор пользователя, временную метку, сетевой адрес устройства, тип действия, контрольную сумму файла:
    • 01.02 10:00 — Иванов И.И. (Инициатор) загрузил version_1.docx (контрольная сумма: A12F...)
    • 01.02 14:20 — Петрова А. (Юрист) загрузила version_2.docx (контрольная сумма: B34C...). Комментарий: «Поправил п.3.2»
    • 02.02 09:00 — Сидоров (Согласующий) подписал Квалифицированной Электронной Подписью. Контрольная сумма: B34C... Сертификат: №567
    • 03.02 11:00 — Документ направлен Контрагенту через оператора электронного документооборота. Квитанция: №890
    • 04.02 15:30 — Контрагент подписал встречной Квалифицированной Электронной Подписью. Контрольная сумма: B34C...
  3. Юрист проверяет подпись: Модуль электронной подписи обращается к криптопровайдеру через протокол онлайн-проверки статуса сертификата на момент подписания (по сохранённой метке времени) и отображает результат: «Подпись действительна» или «Подпись недействительна» с указанием причины.
  4. При необходимости Юрист скачивает конкретную версию файла из Репозитория документов по контрольной сумме для сверки.
  5. Юрист экспортирует Акт аудита в формате переносимого документа, заверенный системной печатью платформы, содержащий: полную ленту событий, контрольные суммы файлов, данные подписей.

Результат

  1. Сформирован юридически значимый отчёт, доказывающий цепочку событий, с криптографически защищённой лентой аудита и проверенными подписями.

Исключения

  1. Попытка подделать записи в базе данных 1.1. Журнал аудита использует криптографическое связывание записей — каждая запись содержит контрольную сумму предыдущей. Любое изменение нарушает криптографическую цепочку и обнаруживается автоматической проверкой целостности (не реже 1 раза в сутки). При обнаружении нарушения система отправляет оповещение Администратору и ответственному за информационную безопасность.

  2. Сертификат подписи отозван 2.1. Модуль электронной подписи проверяет статус через протокол онлайн-проверки статуса сертификата. Если сертификат отозван после подписания — документ действителен (метка времени подтверждает, что на момент подписания сертификат был действующим). Если отозван до подписания — документ ничтожен, журнал аудита фиксирует нарушение.
